# pm-evaluate

Score a PM interview answer against the framework rubric defined in the question's plugin, and write a detailed feedback file.

The full evaluation logic — rubric parsing, scoring, failure-mode detection, feedback file template — lives in `<root>/_core/evaluator.md`. Read that file before you start. This skill defines the **flow** (where to look, how to load inputs, where to save), the existing `_core/evaluator.md` defines the **content** of evaluation.

## Resolving the repo root

Same as `.claude/skills/pm-question/SKILL.md` — find the directory containing `config.yaml`.

## Step 1: Locate the question being answered

Figure out which question file the user is answering:

1. If the user provides a path to a question file, use that.
2. If a question was generated earlier in this conversation (by `pm-question` or `pm-interview`), use that one.
3. If only a category is given, find the most recent `ON_DEMAND-DAY-*.md` in `<root>/questions/<category>/` that doesn't already have a matching answer file. If none exists, fall back to the most recent `DAY-MM-DD.md`.
4. If you still can't tell, ask: "Which question are you answering? Paste the path, or tell me the category."

## Step 2: Load the framework

Read the question file's frontmatter. Extract `framework_path`.

Read the framework file at that path. Parse:
- `## The [N] Steps` — interview structure
- `## Evaluation Rubric (Total: 10 points)` — the rubric (component-by-component)
- `## Common Failure Modes` — patterns to detect in the user's answer
- `## Scoring Interpretation` — score ranges (Excellent / Strong / Decent / Weak / Poor)

Validate the rubric components sum to 10. If not, note the deviation in chat and proceed.

## Step 3: Load the user's answer

The user provides their answer in one of three formats. Detect which and load accordingly.

### A. Pasted text

Use the text directly. If the user pasted only a short reply ("done") with no answer content, ask them to actually paste the answer.

### B. Local file path

If the user gave a path that looks like a local file (`/Users/...`, `~/...`, `./...`, or just a filename in the cwd), use the Read tool to load it.

### C. Google Drive link

If the user shared a Drive link (`drive.google.com/file/d/<FILE_ID>/...` or `docs.google.com/document/d/<FILE_ID>/...`), try in this order:

1. **Public-file fetch via WebFetch.** Convert the share link to a direct download URL:
   - `https://drive.google.com/file/d/<FILE_ID>/view?...` → `https://drive.google.com/uc?export=download&id=<FILE_ID>`
   - For Google Docs: `https://docs.google.com/document/d/<FILE_ID>/export?format=txt`
   
   Use WebFetch. If the response is the actual file content, you're done. If it's an HTML page containing "you need access", "sign in", or a Google login form, the file isn't public.

2. **Drive MCP fallback.** Check the tools available to you for any Google Drive MCP (tool names containing `drive` plus actions like `read_file_content`, `download_file_content`, `get_file_metadata`). If present, use it with the FILE_ID extracted from the link.

3. **Ask the user** if both fail:
   > "I couldn't fetch that file. Options:
   > (a) Re-share it as 'Anyone with the link can view' and resend the link.
   > (b) Download it locally and give me the path.
   > (c) Paste the answer text directly here."

## Step 4: Score against the rubric

Follow the scoring procedure in `<root>/_core/evaluator.md` under "Scoring Process". For each rubric component:

1. Identify the portion of the user's answer that addresses this component.
2. Match against the Excellent / Partial / Missing criteria.
3. Award points and write a one-line justification.

Sum to a total score out of 10.

## Step 5: Detect failure modes

For each entry in the framework's `## Common Failure Modes`, check whether the user's answer exhibits that pattern. Note all matches for the feedback file.

## Step 6: Write the answer file

Filename:
- If the question was generated by a skill (`ON_DEMAND-DAY-MM-DD-HHMM.md`): write `ON_DEMAND-ANSWER-MM-DD-HHMM.md` (mirror the time suffix).
- If the question is a Cowork daily file (`DAY-MM-DD.md`): write `ANSWER-MM-DD.md` (existing convention).

Write to `<root>/questions/<category>/<answer-filename>`. Never overwrite an existing answer — if one exists, suffix with `-v2`, `-v3`, etc.

Use the **Feedback File Template** in `<root>/_core/evaluator.md` as the structure. Frontmatter must include:

```yaml
---
date: YYYY-MM-DD
category: <category-id>
framework: <framework_name>
question_file: <relative path to the question file>
time_taken: <minutes, or null if not tracked>
score: X/10
interpretation: Excellent | Strong | Decent | Weak | Poor
evaluated_by: pm-evaluate
evaluated_at: <ISO 8601 timestamp>
---
```

Body sections per the template in `_core/evaluator.md`:
- Your Answer (transcribed or summarized, organized by framework steps)
- Evaluation Breakdown (component by component, with criteria met/missed and one-line justification)
- Total Score & Interpretation
- Strengths (3+ specific bullets)
- Areas for Improvement (3+ bullets with specific actionable advice)
- Detected Failure Modes (or "none detected")
- Recommendations (3 next steps)

## Step 7: Short summary in chat

In chat, give a brief result — not the full feedback file (which is already on disk):

- Score line: "**6/10 — Decent.**"
- One-line top strength.
- One-line top improvement area.
- Path to the full answer file (repo-relative).

## Notes

- The source of truth for evaluation content (rubric parsing, scoring rules, feedback file template) is `<root>/_core/evaluator.md`. If anything in this skill conflicts with that file, that file wins.
- Don't dump the full feedback in chat. The answer file is the deliverable; the chat summary is just a pointer.
- If the question file's frontmatter is missing or malformed, ask the user which framework they were using and proceed with that.
